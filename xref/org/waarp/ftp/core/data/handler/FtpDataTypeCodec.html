43  @Sharable
44  class FtpDataTypeCodec extends MessageToMessageCodec<DataBlock, DataBlock> {
45      /*
46       * 3.1.1. DATA TYPES Data representations are handled in FTP by a user specifying a
47       * representation type. This type may implicitly (as in ASCII or EBCDIC) or explicitly (as in
48       * Local byte) define a byte size for interpretation which is referred to as the
49       * "logical byte size." Note that this has nothing to do with the byte size used for
50       * transmission over the data connection, called the "transfer byte size", and the two should
51       * not be confused. For example, NVT-ASCII has a logical byte size of 8 bits. If the type is
52       * Local byte, then the TYPE command has an obligatory second parameter specifying the logical
53       * byte size. The transfer byte size is always 8 bits. 3.1.1.1. ASCII TYPE This is the default
54       * type and must be accepted by all FTP implementations. It is intended primarily for the
55       * transfer of text files, except when both hosts would find the EBCDIC type more convenient.
56       * The sender converts the data from an internal character representation to the standard 8-bit
57       * NVT-ASCII representation (see the Telnet specification). The receiver will convert the data
58       * from the standard form to his own internal form. In accordance with the NVT standard, the
59       * <CRLF> sequence should be used where necessary to denote the end of a line of text. (See the
60       * discussion of file structure at the end of the Section on Data Representation and Storage.)
61       * Using the standard NVT-ASCII representation means that data must be interpreted as 8-bit
62       * bytes. The Format parameter for ASCII and EBCDIC types is discussed below. 3.1.1.2. EBCDIC
63       * TYPE This type is intended for efficient transfer between hosts which use EBCDIC for their
64       * internal character representation. For transmission, the data are represented as 8-bit EBCDIC
65       * characters. The character code is the only difference between the functional specifications
66       * of EBCDIC and ASCII types. End-of-line (as opposed to end-of-record--see the discussion of
67       * structure) will probably be rarely used with EBCDIC type for purposes of denoting structure,
68       * but where it is necessary the <NL> character should be used. 3.1.1.3. IMAGE TYPE The data are
69       * sent as contiguous bits which, for transfer, are packed into the 8-bit transfer bytes. The
70       * receiving site must store the data as contiguous bits. The structure of the storage system
71       * might necessitate the padding of the file (or of each record, for a record-structured file)
72       * to some convenient boundary (byte, word or block). This padding, which must be all zeros, may
73       * occur only at the end of the file (or at the end of each record) and there must be a way of
74       * identifying the padding bits so that they may be stripped off if the file is retrieved. The
75       * padding transformation should be well publicized to enable a user to process a file at the
76       * storage site. Image type is intended for the efficient storage and retrieval of files and for
77       * the transfer of binary data. It is recommended that this type be accepted by all FTP
78       * implementations. 3.1.1.4. LOCAL TYPE The data is transferred in logical bytes of the size
79       * specified by the obligatory second parameter, Byte size. The value of Byte size must be a
80       * decimal integer; there is no default value. The logical byte size is not necessarily the same
81       * as the transfer byte size. If there is a difference in byte sizes, then the logical bytes
82       * should be packed contiguously, disregarding transfer byte boundaries and with any necessary
83       * padding at the end. When the data reaches the receiving host, it will be transformed in a
84       * manner dependent on the logical byte size and the particular host. This transformation must
85       * be invertible (i.e., an identical file can be retrieved if the same parameters are used) and
86       * should be well publicized by the FTP implementors. For example, a user sending 36-bit
87       * floating-point numbers to a host with a 32-bit word could send that data as Local byte with a
88       * logical byte size of 36. The receiving host would then be expected to store the logical bytes
89       * so that they could be easily manipulated; in this example putting the 36-bit logical bytes
90       * into 64-bit double words should suffice. In another example, a pair of hosts with a 36-bit
91       * word size may send data to one another in words by using TYPE L 36. The data would be sent in
92       * the 8-bit transmission bytes packed so that 9 transmission bytes carried two host words.
93       * 3.1.1.5. FORMAT CONTROL The types ASCII and EBCDIC also take a second (optional) parameter;
94       * this is to indicate what kind of vertical format control, if any, is associated with a file.
95       * The following data representation types are defined in FTP: A character file may be
96       * transferred to a host for one of three purposes: for printing, for storage and later
97       * retrieval, or for processing. If a file is sent for printing, the receiving host must know
98       * how the vertical format control is represented. In the second case, it must be possible to
99       * store a file at a host and then retrieve it later in exactly the same form. Finally, it
100      * should be possible to move a file from one host to another and process the file at the second
101      * host without undue trouble. A single ASCII or EBCDIC format does not satisfy all these
102      * conditions. Therefore, these types have a second parameter specifying one of the following
103      * three formats: 3.1.1.5.1. NON PRINT This is the default format to be used if the second
104      * (format) parameter is omitted. Non-print format must be accepted by all FTP implementations.
105      * The file need contain no vertical format information. If it is passed to a printer process,
106      * this process may assume standard values for spacing and margins. Normally, this format will
107      * be used with files destined for processing or just storage. 3.1.1.5.2. TELNET FORMAT CONTROLS
108      * The file contains ASCII/EBCDIC vertical format controls (i.e., <CR>, <LF>, <NL>, <VT>, <FF>)
109      * which the printer process will interpret appropriately. <CRLF>, in exactly this sequence,
110      * also denotes end-of-line. 3.1.1.5.2. CARRIAGE CONTROL (ASA) The file contains ASA (FORTRAN)
111      * vertical format control characters. (See RFC 740 Appendix C; and Communications of the ACM,
112      * Vol. 7, No. 10, p. 606, October 1964.) In a line or a record formatted according to the ASA
113      * Standard, the first character is not to be printed. Instead, it should be used to determine
114      * the vertical movement of the paper which should take place before the rest of the record is
115      * printed. The ASA Standard specifies the following control characters: Character Vertical
116      * Spacing blank Move paper up one line 0 Move paper up two lines 1 Move paper to top of next
117      * page + No movement, i.e., overprint Clearly there must be some way for a printer process to
118      * distinguish the end of the structural entity. If a file has record structure (see below) this
119      * is no problem; records will be explicitly marked during transfer and storage. If the file has
120      * no record structure, the <CRLF> end-of-line sequence is used to separate printing lines, but
121      * these format effectors are overridden by the ASA controls.
122      */

187     @Override
188     protected void decode(ChannelHandlerContext ctx, DataBlock msg, List<Object> out) throws Exception {
189         // Is an ASCII or EBCDIC mode or IMAGE mode
190         if (type == TransferType.IMAGE) {
191             out.add(msg);
192             return;
193         } else if (type == TransferType.ASCII || type == TransferType.EBCDIC) {
194             ByteBuf buffer = msg.getBlock();
195             msg.setBlock(decode(buffer));
196             out.add(msg);
197             return;
198         }
199         // Type unimplemented
200         throw new InvalidArgumentException("Type unimplemented in " +
201                 this.getClass().getName() + " codec " + type.name());
202     }

204     /**
205      * 
206      * @param ByteBuf
207      * @return the ByteBuf
208      * @throws Exception
209      */
210     protected ByteBuf decode(ByteBuf ByteBuf)
211             throws Exception {
212         return Unpooled.copiedBuffer(ByteBuf
213                 .toString(type.charset), charsetName);
214     }
215 
216     @Override
217     protected void encode(ChannelHandlerContext ctx, DataBlock msg, List<Object> out) throws Exception {
218         // Is an ASCII or EBCDIC mode or IMAGE mode
219         if (type == TransferType.IMAGE) {
220             out.add(msg);
221             return;
222         } else if (type == TransferType.ASCII || type == TransferType.EBCDIC) {
223             ByteBuf buffer = msg.getBlock();
224             msg.setBlock(encode(buffer));
225             out.add(msg);
226             return;
227         }
228         // Type unimplemented
229         throw new InvalidArgumentException("Type unimplemented in " +
230                 this.getClass().getName() + " codec " + type.name());
231     }

233     /**
234      * 
235      * @param ByteBuf
236      * @return the encoded buffer
237      * @throws Exception
238      */
239     protected ByteBuf encode(ByteBuf ByteBuf)
240             throws Exception {
241         String chString = ByteBuf.toString(charsetName);
242         return Unpooled.copiedBuffer(chString, type.charset);
243     }
